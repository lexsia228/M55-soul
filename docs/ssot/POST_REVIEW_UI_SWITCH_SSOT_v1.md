# POST-REVIEW UI SWITCH SSOT (M55) v1

## 目的
Stripe審査通過後に、提出用の公開トップ/LP（審査専用・最小構成）から、アプリ版プロトタイプUIへ段階的に移行する。
ただし「課金・返金・サポート・法務の整合性」と「誤認リスク（占い扱い等）」を上げないことを最優先とする。

## 重要な前提（審査後も守る不変条件）
- 事業実態は「単発のデジタルレポート販売」を維持する。
- 価格・提供方法・返金・サポート導線・法務ページURLは恒常的に整合していること。
- 公開ページから禁止語彙（占い/鑑定/運勢/予言/開運/霊感/当たる 等）を出さない。
- 新規の決済導線・新プラン表示・煽りコピーは禁止（段階導入は必ずFeature Flag）。
- 購入状態SSOTはDB/PurchaseCacheのみ（増殖しない）。
- 背景NoTouch、通知っぽさ禁止、URL文脈注入禁止を継続。

## リスク評価（現状プロトタイプHTML）
- プロトタイプに「鑑定/タロット/恋愛」等の語彙が含まれる場合、公開トップにそのまま置くと再確認/誤認リスクが上がる。
- 審査後も「顧客が明細で理解できる」「返金/サポートに即到達できる」構造を維持する必要がある。

## 採用する移行戦略（推奨：安全）
### Phase 0（審査通過〜即時）
- /（トップ）と /dtr/lp は現行の“ストアフロント”を維持（価格¥1,000、提供、返金、サポート、法務導線が明確）。
- 変更は原則禁止。差し戻し指示が来た場合のみ最小差分で対応。

### Phase 1（UI拡張の隔離）
- プロトタイプUIは /app（または /prototype）に配置し、ストアフロントからは「中立的な導線」で遷移させる。
- /app 側の公開テキストも禁止語彙を出さない（必要なら言い換え）。
- 購入・返金・サポートの基幹導線は /support /legal/* を継続利用。

### Phase 2（トップ差し替えをする場合の最低条件）
トップをプロトタイプに寄せるなら、以下を必ず満たす：
- 画面上部に「価格¥1,000」「提供（即時閲覧/配送なし）」「返金（/legal/refund）」「サポート（/support）」の固定ブロックを残す。
- 禁止語彙は全面言い換え（例：「鑑定」→「分析レポート」「傾向の整理」）。
- “購入ボタンや決済導線”は増やさない（既存の導線を維持 or Feature Flagで段階導入）。

## Go/No-Go判定
- Go条件：公開HTML上で価格/返金/サポート/法務が常時到達可能、禁止語彙0、誤認表現0、既存決済導線を増やしていない。
- No-Go条件：トップに禁止語彙が出る／価格が曖昧になる／返金・サポート導線が弱くなる／新規決済導線が増える。

## 運用ログ（証跡）
- 変更前後のHTMLは Desktop\stripe審査\evidence\ に保存（repo外）。
- 提出用ページを差し替える前に、同等の機械チェック（price/refund/support/forbidden=0）を再実行してPASSを証跡化する。
